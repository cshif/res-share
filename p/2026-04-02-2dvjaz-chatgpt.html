Question
看起來按下 copy response 完全只能拿到 response 本身。如果我要在按下 copy response 時不只複製 response 還要去 DOM 找距離這則 response 最近的 data-turn-id 、拿到值後去找另一個 data-turn-id + data-turn="user" ？
Answer

對，這樣做是對的，而且比只讀剪貼簿穩很多。

你要的流程應該是：

  1. 使用者按下 Copy response
  2. 你的 content script 在 click 當下 先從 DOM 找到「這顆按鈕所屬的 assistant turn」
  3. 從那個 turn 容器拿 data-turn-id
  4. 再往前找最近一個 data-turn="user" 的 turn
  5. 把這兩個 turn 的資訊，和剪貼簿裡的 response 一起存起來

原因很簡單：navigator.clipboard.readText() 只能拿到剪貼簿裡的文字，不會帶回 DOM 的 data-* 屬性；要拿 data-turn-id，就必須在頁面 DOM 上找。另一方面，content script 雖然和頁面 JS 執行環境隔離，但共享頁面的 DOM 存取，所以這種做法在 extension 裡是可行的。(Chrome for Developers)

不過有一個前提要先講白： data-turn-id / data-turn="user" 不是我找到的官方公開 API 契約。我能找到的是社群公開的 ChatGPT DOM 快照與使用者腳本，顯示某些版本的 ChatGPT 介面確實出現過 data-turn-id 這類屬性；所以它可以拿來用，但應視為不穩定 DOM 細節，不要當成長期保證不變的欄位。(Gist)

建議的找法

最穩的原則

不要只靠「上一個 sibling 一定是 user」。 雖然 previousElementSibling 可以拿到同層級前一個元素，但它只保證是緊鄰前一個 sibling；如果中間插了別的節點、系統訊息或 UI 容器，就可能失準。(MDN Web Docs)

比較穩的方式是：

  • 先用 closest('[data-turn-id]') 找出目前這顆 copy 按鈕屬於哪一個 turn
  • 再抓整個對話區內所有 turn，例如 querySelectorAll('[data-turn-id]')
  • querySelectorAll() 回來的是文件順序的 static NodeList
  • 找到目前 assistant turn 在陣列中的位置
  • 往前掃，找到第一個 data-turn="user" 的 turn

closest() 會從元素自己一路往上找符合 selector 的祖先；querySelectorAll() 會回傳文件順序的匹配元素集合，這正好適合做「往前回找最近上一則 user turn」。(MDN Web Docs)

你應該採用的策略

不建議

  • 按下 copy 後，只讀剪貼簿，再嘗試從 response 文字反推它是哪個 turn

建議

  • 按下 copy 時，同步抓 DOM 上的 turn 關聯

  • 之後再去讀剪貼簿

  • 最後把這三樣東西一起存：

    • assistantTurnId
    • nearestUserTurnId
    • copiedResponseMarkdown

實作範例

下面這版就是你要的邏輯骨架：

document.addEventListener(
  "click",
  async (event) => {
    const btn = event.target.closest("button");
    if (!btn) return;

    const text = [
      btn.getAttribute("aria-label") || "",
      btn.getAttribute("title") || "",
      btn.innerText || "",
      btn.textContent || ""
    ].join(" ").toLowerCase();

    if (!text.includes("copy")) return;

    // 1. 找這顆按鈕所屬的 assistant turn
    const assistantTurnEl = btn.closest("[data-turn-id]");
    if (!assistantTurnEl) return;

    const assistantTurnId =
      assistantTurnEl.dataset.turnId ||
      assistantTurnEl.getAttribute("data-turn-id");

    // 2. 找整個對話裡所有 turn（依文件順序）
    const allTurns = [...document.querySelectorAll("[data-turn-id]")];

    // 3. 找目前 assistant turn 的索引
    const assistantIndex = allTurns.indexOf(assistantTurnEl);
    if (assistantIndex === -1) return;

    // 4. 往前找最近的 user turn
    let nearestUserTurnEl = null;
    for (let i = assistantIndex - 1; i >= 0; i--) {
      const el = allTurns[i];
      const turnType = el.getAttribute("data-turn") || el.dataset.turn;
      if (turnType === "user") {
        nearestUserTurnEl = el;
        break;
      }
    }

    const nearestUserTurnId = nearestUserTurnEl
      ? nearestUserTurnEl.dataset.turnId ||
        nearestUserTurnEl.getAttribute("data-turn-id")
      : null;

    // 5. 再去讀剪貼簿
    const copiedText = await navigator.clipboard.readText();

    console.log({
      assistantTurnId,
      nearestUserTurnId,
      copiedText
    });

    // 6. 存本地 / 傳給 sw.js
    chrome.runtime.sendMessage({
      type: "CREATE_LOCAL_DRAFT",
      payload: {
        assistantTurnId,
        nearestUserTurnId,
        copiedText,
        sourceUrl: location.href,
        copiedAt: new Date().toISOString()
      }
    });
  },
  true
);

這段用到的 API 都是標準 DOM 能力： closest() 找最近祖先、dataset 讀 data-*、querySelectorAll() 取匹配元素清單，而且元素順序是文件順序。(MDN Web Docs)

兩個實作提醒

第一個，把 selector 寫成可替換的 helper。 因為 data-turn-id 很可能哪天就改名。你可以先集中寫成：

function getTurnElementFromCopyButton(button) {
  return button.closest("[data-turn-id]") || button.closest("article");
}

這樣以後 ChatGPT DOM 改版，只要改這個 helper。

第二個，不要假設 every assistant turn 前面一定立刻是一個 user turn。 如果你只用 previousElementSibling，很容易被中間插入的非 turn 元素搞壞；用「抓全部 turn，再往前掃第一個 user turn」比較耐用。previousElementSibling 只回傳同層的前一個元素，不會幫你跳過不相干結構。(MDN Web Docs)

一句話結論

可以，而且正確做法就是：在 Copy response 的 click 當下，先從該按鈕往上抓最近的 assistant data-turn-id，再沿 turn 列表往前找最近的 data-turn="user"；不要想從剪貼簿內容本身反推。 不過這套做法建立在 ChatGPT 當前 DOM 結構上，data-turn-id 應視為可用但不穩定的非官方欄位。(Gist)
